## 1. 项目概述

**FormatPass** 是一个纯前端的 Web 工具，帮助大学生检测毕业论文/期刊论文的 Word 格式是否符合学术规范。

- **部署地址：** https://formatpass.vercel.app
- **GitHub：** https://github.com/Tunnello/FormatPass
- **架构：** 客户端处理全部逻辑（解压 docx → 解析 XML → 规则比对 → 生成报告），Vercel 仅做静态托管

### 3.3 DOCX 解析层 (`lib/parser/`)

**`docx-unzip.ts`** — 使用 JSZip 解压 docx

```typescript
import JSZip from 'jszip'

export interface DocxFiles {
  documentXml: string
  stylesXml?: string
}

export async function unzipDocx(arrayBuffer: ArrayBuffer): Promise<DocxFiles> {
  const zip = await JSZip.loadAsync(arrayBuffer)
  const documentXml = await zip.file('word/document.xml')?.async('string')
  const stylesXml = await zip.file('word/styles.xml')?.async('string')
  if (!documentXml) throw new Error('无法找到 word/document.xml')
  return { documentXml, stylesXml }
}
```

**`xml-extractor.ts`** — 解析 document.xml 和 styles.xml

核心命名空间：`http://schemas.openxmlformats.org/wordprocessingml/2006/main`

提取内容：
- `ParagraphData`：文本、样式 ID、大纲级别、段落属性 (pPr)、文本运行属性 (rPr)
- `PageSetup`：宽/高（twips）、方向、页边距（twips）
- `StyleData`：样式名、继承关系 (basedOn)、段落/文本运行属性

**`style-resolver.ts`** — 样式继承解析

解析优先级：**直接属性 > 命名样式 (pStyle) > 文档默认样式 (docDefaults)**

导出两个函数：
- `resolveEffectiveStyle`：解析字体、字号、加粗、斜体（从 rPr 链）
- `resolveParagraphStyle`：解析对齐、段前段后间距、行距（从 pPr 链）

```typescript
export interface EffectiveStyle {
  fontSize: number | null      // half-points
  fontAscii: string | null
  fontEastAsia: string | null
  bold: boolean | null
  italic: boolean | null
  alignment: string | null
  spaceBefore: number | null   // twips
  spaceAfter: number | null    // twips
  lineSpacing: { value: number; rule: 'auto' | 'exact' } | null
}
```

### 3.4 规则引擎 (`lib/engine/`)

**`rule-types.ts`** — 核心类型定义

```typescript
interface FormatRule {
  id: string
  category: 'page' | 'body' | 'heading'
  name: string           // 人类可读，如"页边距-上"
  target: RuleTarget
  expected: RuleValue
  actual?: RuleValue
  status: 'pending' | 'pass' | 'fail'
  location?: string      // 失败位置提示
}

type RuleTarget =
  | { type: 'page-margin'; side: 'top' | 'bottom' | 'left' | 'right' }
  | { type: 'page-size'; dimension: 'width' | 'height' }
  | { type: 'page-orientation' }
  | { type: 'font'; scope: 'body' | 'heading'; level?: number; script: 'ascii' | 'eastAsia' }
  | { type: 'font-size'; scope: 'body' | 'heading'; level?: number; script: 'ascii' | 'eastAsia' }
  | { type: 'paragraph-spacing'; scope: 'body' | 'heading'; level?: number; kind: 'before' | 'after' | 'line' }
  | { type: 'alignment'; scope: 'body' | 'heading'; level?: number }

type RuleValue =
  | { kind: 'length'; value: number; unit: 'cm' | 'pt' | 'twip' }
  | { kind: 'string'; value: string }
  | { kind: 'enum'; value: string }
  | { kind: 'lineSpacing'; value: number; rule: 'auto' | 'exact' }
```

**`rule-builder.ts`** — 表单数据 → 规则数组

接收 `RuleFormData`，为每个可检测项生成一条 `FormatRule`：
- 页面设置：4 条页边距 + 2 条纸张尺寸 + 1 条方向
- 正文格式：6 条（中文字体、西文字体、字号、行距、段前、段后）
- 标题格式：每级 5 条（字体、字号、段前、段后、对齐）

**`rule-checker.ts`** — 核心比对逻辑

```typescript
export function checkRules(
  rules: FormatRule[],
  doc: ExtractedDocument,
  styles: Map<string, StyleData>
): FormatRule[]
```

比对逻辑：
1. **页面规则**：直接从 `doc.pageSetup` 读取实际值
2. **段落规则**：找到目标段落 → `resolveEffectiveStyle` / `resolveParagraphStyle` → 提取实际值
3. **数值比对**：预期值与实际值转换到同一单位后比较差值
   - cm：容差 `±0.06`
   - pt：容差 `±0.6`
   - 行距倍数：容差 `±0.05`
4. **字符串比对**：字体名不区分大小写、忽略前后空格

标题段落识别策略（按优先级）：
1. 样式名为 `Heading1` / `Heading2` / `Heading3`
2. `outlineLvl` 属性为 0/1/2
3. 启发式兜底：编号像标题（如"1.1"）且加粗/大号字体

### 3.5 报告导出 (`lib/exporter/`)

**`format-value.ts`** — 共享格式化工具

```typescript
export function formatValue(v: RuleValue | undefined): string {
  if (!v) return '未检测到'
  if (v.kind === 'length') return `${v.value} ${v.unit}`
  if (v.kind === 'lineSpacing') return v.rule === 'auto' ? `${v.value} 倍` : `${v.value} 磅`
  if (v.kind === 'enum') return String(v.value)
  return String(v.value)
}
```

**`html-exporter.ts`** — 生成独立 HTML 文件

- 内嵌 CSS（基于 Clay Design 颜色）
- 分类卡片展示，绿/红颜色标记
- 所有用户输入值经过 HTML 转义（防 XSS）

**`docx-exporter.ts`** — 生成 Word 文件

- 使用 `docx` 库的 `Document` + `Packer.toBlob`
- 标题、摘要、分类别表格
- 结果列单元格背景色：绿色通过 / 红色失败

### 3.6 单位转换 (`lib/utils/unit-converter.ts`)

Word 内部单位与常用单位互转：

| 函数 | 公式 |
|---|---|
| `twipToCm` | `twip * 2.54 / 1440` |
| `cmToTwip` | `Math.round(cm * 1440 / 2.54)` |
| `twipToPt` | `twip / 20` |
| `ptToTwip` | `Math.round(pt * 20)` |
| `halfPointToPt` | `hp / 2` |
| `ptToHalfPoint` | `Math.round(pt * 2)` |

## 4. 数据流

```
用户上传 .docx
    ↓
JSZip 解压 → document.xml + styles.xml
    ↓
DOMParser 解析 XML → ExtractedDocument + Style Map
    ↓
RuleBuilder 将表单数据转为 FormatRule[]
    ↓
RuleChecker 比对实际值与预期值 → 标记 pass/fail
    ↓
ReportView 展示结果
    ↓
ReportExport 生成 HTML / DOCX 下载
```

## 6. 开发规范

- **TDD：** 核心模块均先写测试再实现（unit-converter, xml-extractor, style-resolver, rule-builder, rule-checker, html-exporter, docx-exporter）
- **类型安全：** 无 `any`，全部使用显式类型
- **组件拆分：** 每个文件单一职责，共享组件提取到 `components/`
- **样式：** Tailwind 优先，避免 inline style（动态颜色用 `className` + 条件或 Tailwind 任意值）
- **无障碍：** `aria-expanded`, `aria-controls`, `aria-invalid`, `aria-describedby`, `sr-only`

---

## 7. 已知限制

1. **仅支持 `.docx`**：`.doc` 是二进制 OLE 格式，浏览器端无成熟解析库
2. **字体名映射：** Word 内部以英文记录字体（如 `SimSun` = 宋体），系统维护常见中英文字体映射表
3. **样式继承覆盖 95% 场景：** 极端自定义样式可能识别偏差
4. **标题识别依赖样式规范：** 鼓励用户使用 Word 内置「标题 1/2/3」样式；纯手动格式化的标题采用启发式识别
5. **仅检测第一个目标段落：** 正文/标题只取第一个段落比对，不遍历全文
