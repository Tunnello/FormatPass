# 毕业论文/期刊论文格式检测工具 — 阶段一设计文档

**Date:** 2026-05-30  
**Scope:** Phase 1 (MVP)  
**Deployment Target:** Vercel  
**Architecture:** Client-side only (browser processing)

---

## 1. 项目概述

一个面向在校大学生、研究生、博士生的 Web 工具，帮助用户检测毕业论文或期刊会议论文的 Word 格式是否符合学术规范。

**核心价值：** 用户上传 `.docx` 论文，手动填写格式规则，浏览器内即时完成解析与检测，输出文字版合规报告，并支持导出为 HTML 或 DOCX。

---

## 2. 阶段一范围（In Scope / Out of Scope）

### 2.1 In Scope（本期必做）
- 上传并解析 `.docx` 文件（不支持 `.doc` 老格式）
- 用户通过表单手动填写格式规则
- 检测以下格式维度：
  - **页面设置**：页边距（上/下/左/右）、纸张大小（A4/Letter）、页面方向（纵向/横向）
  - **正文格式**：中文字体、西文字体、字号、行距、段前段后间距
  - **标题格式**：一级/二级/三级标题的字体、字号、段前段后、对齐方式
- 生成文字版检测报告（分类展示 ✅ 合格 / ❌ 不合格）
- **导出检测报告**：HTML 格式、DOCX 格式
- 支持修改规则后重新检测，无需重新上传文件

### 2.2 Out of Scope（后续阶段实现）
- 上传模板文档自动提取规则
- 预置高校/期刊标准库
- 页面级红绿视觉标注（类似查重报告的逐页定位）
- 一键自动修正格式
- 导出修改后的 Word / PDF
- 支持 `.doc` 格式
- 图表/公式/参考文献/目录格式检测

---

## 3. 用户流程

```
首页（上传论文）
    ↓
规则填写页（手动选择/输入格式规则）
    ↓
检测中（Web Worker 解析与比对，显示进度）
    ↓
报告页（文字版结果 + 导出按钮）
    ↓
（可选）返回修改规则 → 重新检测
```

**设计原则：极简输入。** 每个字段优先使用下拉选择器（Select）而非自由文本输入，降低用户认知成本。

## 4. 技术架构

### 4.1 架构模式：纯前端处理

所有计算在浏览器内完成，Vercel 仅托管静态 Next.js 应用。

```
┌─────────────────────────────────────────────────────┐
│                  Next.js (Vercel)                   │
│  ┌─────────┐   ┌──────────┐   ┌─────────────────┐  │
│  │ Upload  │ → │  Parser  │ → │  Rule Engine    │  │
│  │   UI    │   │(JSZip+XML│   │  (比对逻辑)      │  │
│  └─────────┘   └──────────┘   └────────┬────────┘  │
│                                         ↓           │
│                              ┌─────────────────┐    │
│                              │ Report + Export │    │
│                              │   (HTML/DOCX)   │    │
│                              └─────────────────┘    │
└─────────────────────────────────────────────────────┘
              ↑ 零后端请求，零服务器成本
```

### 4.2 核心数据流

1. **文件读取**：用户选择 `.docx` → `FileReader` 读为 `ArrayBuffer`
2. **解压**：`JSZip` 解压 docx，提取内部 XML 文件
3. **解析**：
   - `word/document.xml` → 段落结构、文本运行（run）、页面设置（sectPr）
   - `word/styles.xml` → 命名样式定义（如 `Heading1`、`Normal`）
   - 关联样式继承链（段落直接属性 > 命名样式 > 默认样式）
4. **规则比对**：将提取的实际格式与用户填写的规则逐项比较
5. **生成报告**：标记每条规则为 `pass` / `fail`，`fail` 时记录预期值、实际值、位置提示
6. **渲染与导出**：展示报告，提供 HTML / DOCX 下载

### 4.3 技术栈

| 层级 | 技术 | 说明 |
|---|---|---|
| 框架 | Next.js 14 (App Router) | Vercel 原生支持，静态导出 |
| 语言 | TypeScript | 类型安全，便于维护解析逻辑 |
| 样式 | Tailwind CSS | 原子化样式，快速实现 Clay Design |
| 解压 | `jszip` | 浏览器端解压 ZIP，成熟稳定 |
| XML 解析 | 原生 `DOMParser` | 零依赖，浏览器内置 |
| DOCX 生成 | `docx` (npm) | 客户端生成 Word 报告 |
| 状态管理 | React `useReducer` | 阶段一状态复杂度可控，无需外部库 |
| 多线程 | Web Workers | 解析大文件时避免阻塞 UI |

## 6. 规则引擎设计

### 6.1 规则数据结构

```typescript
interface FormatRule {
  id: string;
  category: 'page' | 'body' | 'heading';
  name: string;               // 人类可读名称，如"正文西文字体"
  target: RuleTarget;         // 检测对象定义
  expected: RuleValue;        // 用户填写的预期值
  actual?: RuleValue;         // 解析后得到的实际值
  status: 'pending' | 'pass' | 'fail';
  location?: string;          // 失败位置提示，如"第 3 章标题"
}

type RuleTarget =
  | { type: 'page-margin'; side: 'top' | 'bottom' | 'left' | 'right' }
  | { type: 'page-size'; dimension: 'width' | 'height' }
  | { type: 'page-orientation' }
  | { type: 'font'; scope: 'body' | 'heading'; level?: number; script: 'ascii' | 'eastAsia' }
  | { type: 'font-size'; scope: 'body' | 'heading'; level?: number; script: 'ascii' | 'eastAsia' }
  | { type: 'paragraph-spacing'; scope: 'body' | 'heading'; level?: number; kind: 'before' | 'after' | 'line' }
  | { type: 'alignment'; scope: 'body' | 'heading'; level?: number };

type RuleValue =
  | { kind: 'length'; value: number; unit: 'cm' | 'pt' | 'twip' }
  | { kind: 'string'; value: string }
  | { kind: 'enum'; value: string }
  | { kind: 'lineSpacing'; value: number; rule: 'auto' | 'exact' }; // auto = 倍数, exact = 固定磅值
```

### 6.2 检测逻辑

对每条规则执行以下步骤：

1. **定位目标段落/样式**
   - `page-margin` / `page-size` / `page-orientation` 目标：读取 `document.xml` 中的 `sectPr`（节属性），`pgSz` 元素同时包含宽高和方向
   - `body` 目标：遍历所有未明确标记为标题的段落（或通过样式名判断为 `Normal` / `Body Text`）
   - `heading` 目标：通过样式名（`Heading1` / `Heading2` / `Heading3`）或大纲级别（`outlineLvl`）定位

2. **提取实际值**
   - 解析段落属性（`pPr`）和文本运行属性（`rPr`）
   - 处理样式继承：直接属性 > 命名样式（`pStyle`）> 文档默认样式（`docDefaults`）
   - 单位转换：Word 内部使用 `twips`（1/20 pt）和 `half-points`，统一转换为人类可读单位（cm / pt）

3. **比对与容错**
   - 长度类规则：允许 `±0.05 cm` 或 `±0.5 pt` 误差（Word 转换精度问题）
   - 字符串类规则（字体名）：不区分大小写，忽略前后空格
   - 枚举类规则（对齐方式）：精确匹配

4. **记录结果**
   - `pass`：实际值在容错范围内符合预期
   - `fail`：记录 `expected`、`actual`、`location`（取第一个不符的段落位置）

### 6.3 标题识别策略

Word 中标题可能通过以下方式标记，按优先级识别：
1. 明确应用了样式 `Heading1` / `Heading2` / `Heading3`
2. 样式中设置了 `outlineLvl`（大纲级别）为 0/1/2
3. 段落编号看起来像标题（如 "1." / "1.1" / "1.1.1"）且应用了加粗/大号字体 —— **此条为启发式兜底，需注明可能不准确**

阶段一中，若用户论文未使用标准标题样式（纯手动加粗+改字号），系统会尝试兜底识别，并在报告中提示「未检测到标准标题样式，采用启发式识别，结果仅供参考」。

## 7. 报告与导出

### 7.1 浏览器内报告展示

报告按三个类别分组，每组内按规则 `id` 排序。合格项默认折叠（或默认展开，根据屏幕空间决定），不合格项始终高亮展示。

每条检测结果展示：
- 状态图标（✅ / ❌）
- 规则名称
- 实际值（合格时）或「预期 X，实际 Y」（不合格时）
- 位置提示（不合格时）

### 7.2 HTML 导出

生成独立的 `.html` 文件，包含：
- 内嵌 CSS（基于 Tailwind 颜色令牌，绿 `#22c55e` / 红 `#ef4444`）
- 报告头部：检测时间、论文文件名
- 与网页版相同的分类展示结构
- 通过 `Blob` + `URL.createObjectURL` 触发下载

### 7.3 DOCX 导出

使用 `docx`（npm 库）在浏览器内生成 Word 文件，包含：
- 标题：「格式检测报告」
- 信息栏：论文文件名、检测时间
- 汇总表：合格项数 / 不合格项数
- 详情表格：规则名称 | 预期值 | 实际值 | 结果
  - 结果列单元格背景色：绿色（通过）/ 红色（不通过）
- 通过 `Blob` + `URL.createObjectURL` 触发下载

## 8. 错误处理与边界情况

| 场景 | 行为 |
|---|---|
| 上传非 `.docx` 文件 | 上传区域显示红色提示「请上传 .docx 格式的文件」，阻止进入下一步 |
| 文件损坏或无法解压 | 显示错误提示「文件解析失败，请检查文件是否完整」，提供「重新上传」按钮 |
| 文档缺少 `styles.xml` | 跳过样式继承解析，仅使用直接属性检测；报告中提示「未检测到样式定义」 |
| 文档无标准标题样式 | 启用启发式标题识别；报告中增加提示条「未使用标准标题样式，结果仅供参考」 |
| 浏览器内存不足（>30MB） | 上传前检查文件大小，超过 30MB 时提示「文件过大，建议拆分为章节分别检测」 |
| 解析耗时过长 | Web Worker 中每解析 100 个段落发送一次进度消息；UI 显示「正在解析第 X 页…」 |
| 用户刷新/关闭页面 | 无影响，纯前端无持久化状态；重新上传即可 |

## 11. 已知限制与假设

1. **仅支持 `.docx`**：`.doc` 是二进制 OLE 格式，浏览器端无成熟解析库，阶段一明确放弃。
2. **字体名检测受系统字体影响**：Word 中字体名以英文记录（如 `SimSun` = 宋体），系统维护映射表处理常见中英文字体名。
3. **复杂样式继承可能有遗漏**：Word 样式系统非常复杂（基于样式、链接样式、默认样式、直接格式化），阶段一覆盖 95% 常见论文场景，极端自定义样式可能识别偏差。
4. **标题识别依赖样式规范**：鼓励用户使用 Word 内置「标题 1/2/3」样式；纯手动格式化的标题采用启发式识别，准确性有限。
5. **DOCX 生成不保留原文档内容**：导出的 DOCX 报告是新建的检测结果文件，不是在原论文上标注。
